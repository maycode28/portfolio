치기공소 공정 관리의 한계를 해결하는 실시간 보철물 위치 트래킹 시스템

기존 전산 프로그램이 해결하지 못하던 보철물의 실시간 위치 추적과 작업자별 부하 예측 문제를 현업 인터뷰를 통해 날카롭게 정의하고, 이를 데이터 구조로 변환하여 해결해 나간 데이터 중심의 관리 시스템입니다.

code GitHub 저장소

문제를 정의하고 해결한 과정

  • 현업 시절의 페인 포인트를 개발 지식으로 재해석하여 문제 정의 캐나다 치기공소 재직 당시 매니저와 고민했으나 당시에는 기술적 한계로 풀지 못했던 실무 이슈를, 개발 공부를 시작한 후 주도적으로 다시 추적하여 구체화했습니다. 기존 전산 프로그램이 있음에도 늘 발생하던 '실시간 위치 파악 불가능'과 '작업자 부하 예측 불가'라는 현업의 문제를 도출해 냈습니다.
  • 흐름을 통제하는 명확한 상태와 이력 설계 기공물이 단순히 접수되고 끝나는 것이 아니라, 여러 단계의 공정을 거치고 언제든 이전 단계로 돌아갈 수도 있는 동적인 흐름을 안전하게 기록하는 데이터 구조 설계
  • 화면 설계(Figma)와 DB 모델링의 유기적 연결 이메일 피드백에서 요구된 필수 정보(Patient name, Pan #, Shade 등)를 누락 없이 채우기 위해, Figma 화면을 먼저 설계해 보며 가변적인 기공물 데이터 필드를 ERD 구조에 꼼꼼히 반영

Tech Stack

요구사항 분석 기반 데이터 모델링, 도메인 비즈니스 로직 구체화

프로젝트의 시작

개발을 무작정 시작하기 전, 이전 직장(캐나다 치기공소 Shanto Dental)의 랩 매니저 Julia와 이메일을 주고받으며 기존 소프트웨어가 해결해주지 못해 현업에서 겪고 있는 진짜 문제점들이 무엇인지 수집하고 분석했습니다.


💡 이메일에서 도출한 핵심 요구사항 요약
  • 기존 소프트웨어(EasyRx)는 주문 접수와 인보이스 발행에는 특화되어 있으나, 제작 과정의 동적인 트래킹이 안 됨.
  • 기술자들이 작업 단계마다 'In/Out'을 명확히 체크하지 않고 완료할 때만 스캔하기 때문에, 현재 기공물이 물리적으로 어디 있는지 추측에 의존함.
  • 특정 날짜나 특정 기술자에게 일이 몰리는 병목 현상을 미리 예측할 수 없어 생산 계획 수립이 어려움.

문제 정의와 기술적 분석

CASE 01

기존 프로그램의 한계 극복: 정적 기록에서 실시간 상태 추적 시스템으로의 전환

[문제 정의] 기존 기공소에서는 전산 프로그램이 아예 없었던 것이 아닙니다. 주문을 등록하고 인보이스를 발행하는 프로그램이 있었지만, 이는 단순 기록과 정적 보고서에 가깝다는 한계가 있었습니다. 작업물이 접수된 후 실제로 [스캔 - 디자인 - 3D 프린팅 - 빌드 - 배송]으로 이어지는 단계적 흐름 속에서 지금 당장 어느 단계에 멈춰 서 있는지, 어디서 병목이 터졌는지는 실시간으로 트래킹되지 않아 매번 기공물 상자(Pan)를 직접 찾으러 다녀야 했습니다. 저는 이 문제를 단순 CRUD 게시판이 아닌, 유연하게 변화하는 상태와 타임라인 관리의 문제로 정의했습니다.

[해결 방식] 메인 테이블에 단순히 현재 상태를 문자열이나 단일 코드 필드로만 덮어쓰면, 도중에 작업이 실패해서 이전 단계로 되돌아가는 반려 상황이 발생했을 때 역추적할 길이 막히게 됩니다. 이를 해결하기 위해 기공물 레코드(LabCase)와 별개로, 상태가 변할 때마다 스냅샷을 찍어 기록하는 상태 이력 테이블(LabCaseProcess)을 1:N 관계로 분리 설계했습니다. 상태가 변경되는 시점, 담당 작업자를 매번 누적하여 저장하도록 강제함으로써, 기존 소프트웨어가 주지 못했던 '실시간 위치 트래킹'과 '공정 지연 분석 데이터'를 완벽히 확보할 수 있는 토대를 만들었습니다.

CASE 02

화면 설계(Figma)를 통한 비즈니스 규칙 정의와 DB 누락 예방

[문제 정의] 제가 경험했던 치기공 지시서는 일반적인 쇼핑몰 주문서와 완전히 달랐습니다. 환자 이름, 의사 이름뿐만 아니라 보철물 종류(크라운, 임플란트 가이드 등), 상악/하악 구분, 치아 번호, 그리고 기공물이 담기는 상자 번호(Pan #)와 색상 정보까지 촘촘하게 엮여 있었습니다. 도메인 분석 없이 테이블을 마구잡이로 짜면, 특정 보철물 종류에서는 쓰이지 않는 수많은 컬럼들이 무더기로 비어있게 되거나, 정규화가 무너지는 문제가 눈에 선했습니다. 저는 코드를 작성하기 전에 사용자의 화면 입력 흐름을 완벽히 파악해야 데이터 제약 조건을 제대로 세울 수 있다고 정의했습니다.

[해결 방식] 실제 사용자가 정보를 입력할 폼 레이아웃을 Figma로 꼼꼼하게 먼저 시각화해 보았습니다. 화면 UI를 이메일 요구사항 요약과 대조하면서, 공통으로 들어갈 데이터(환자명, 마감일, 고유 케이스 번호)와 보철물 종류에 따라 선택적으로 채워지는 가변 데이터를 명확하게 구분해냈습니다. 이 화면 흐름을 바탕으로 데이터베이스 설계 단계에서 다형성을 고려한 스펙 테이블들을 유기적으로 분리시켰습니다. 덕분에 테이블 정규화 수준을 크게 높일 수 있었고, 필수 데이터가 누락되는 휴먼 에러를 DB 제약 조건 레벨에서 튼튼하게 방어해 낼 수 있었습니다.

배운 점 및 성장

architecture 설계가 먼저라는 소프트웨어 개발 원칙의 가치

비즈니스 요구사항과 실제 데이터의 흐름을 꼼꼼하게 쪼개보지 않고 코딩부터 시작하면 나중에 고치는 비용이 훨씬 더 크다는 소중한 교훈을 얻었습니다. 화면 기획과 DB 제약 조건을 미리 맞춰보는 탄탄한 설계 과정이, 복잡한 로직을 개발할 때 흔들리지 않게 지탱해 주는 나침반이 된다는 것을 배운 프로젝트였습니다.

translate 진짜 문제를 찾아 기술로 풀어내는 주도적인 소통 능력

단순히 주어진 스펙대로만 개발하는 사람이 아니라, 현업의 흐름을 직접 분석하고 한 단계 더 들어가 질문을 던지며 시스템을 구상해 나가는 문제 정의형 주니어 개발자로 성장할 수 있었습니다. 특히 아날로그적 틈새와 기존 전산 프로그램 사이의 간극을 포착하여 정형화된 데이터 모델로 변환시키는 전 과정을 홀로 조율하며 큰 성취감과 성장을 경험했습니다.
