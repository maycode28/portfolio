oopsLog

실패를 기록하는 순간, 생각이 바뀌는 경험

oopsLog는 사용자의 실패 경험을 AI를 통해 분석하여 인지 왜곡을 구조적으로 교정하는 서비스입니다. 단순한 감정 기록이 아니라, 인지행동치료(CBT) 기법을 기반으로 사고 방식 자체를 재구성하는 시스템입니다.

주요 역할 및 기여

  • AI 프롬프트 엔지니어링 및 데이터 구조화 설계 CBT 기반 3단계(사실 추출, 왜곡 탐지, 인지 재구성) 분석 프롬프트를 설계하고, 토큰 최적화를 위해 JSON Strict Output의 라벨 압축 규격을 조율
  • 안정적인 AI 연동 및 백엔드 서버 개발 Gemini와 OpenAI를 연동해 한쪽 AI가 작동하지 않을 때 즉시 다른 AI가 대체하는 안정망을 구현하고, 서버 주소가 분리된 분산 환경에서 로그인 세션이 안전하게 유지되도록 함
  • CBT 인터랙티브 사용자 경험 설계 주도 텍스트 위주의 단순 위로를 탈피하고 사용자가 사고 왜곡을 직관적으로 인지할 수 있도록 단계별 비주얼 가이드라인 및 감성 인터랙션 기획

Tech Stack

Spring Boot MariaDB React TypeScript Vite Gemini API OpenAI API Railway Vercel

Key Libraries

JPA / Hibernate, Axios, JSON Strict Output

문제 해결과 기술적 고민

CASE 01

출력 라벨 약어화를 통한 LLM API 토큰 비용 최적화와 스키마 설계 결함

[증상] 구조화된 데이터를 확보하기 위해 AI 분석 API를 반복 호출하는 과정에서 출력 JSON의 장황한 Key값들로 인해 누적 토큰량이 급증하고 인프라 호출 비용이 과다하게 청구되는 현상이 발생했습니다.

[원인 추적 및 해결] 시스템 프롬프트가 반환해야 하는 JSON 규격 내의 'rational_thinking', 'cognitive_distortion_type' 같은 영문 Key 명칭 자체가 응답 바이트의 상당수를 차지함을 파악했습니다. 이를 해결하기 위해 JSON Strict Schema 구조를 전면 수정하여, 출력을 'rt', 'cdt'와 같은 정밀한 압축 약어 라벨로 변경했습니다. 백엔드에서 이를 역직렬화 맵핑 객체로 수신하도록 파이프라인을 조율함으로써, 전체 API 호출 트래픽 비용을 절감하는 성과를 거두었습니다.

[예외 및 로직 고도화] 라벨을 지나치게 축약하는 과정에서 초기 테스트 시 LLM이 문맥적 의미를 혼동하여 다른 약어 자리에 오답을 채워 넣는 예외가 발생했습니다. 이를 방지하기 위해 시스템 프롬프트의 Description 메타데이터 제약을 더욱 꼼꼼하게 다듬고, 백엔드 수신 레이어에 안정적인 이관 가이드라인 및 예외 처리 구문을 구축하여 데이터 무결성을 보장했습니다.

CASE 02

배포 환경 CORS/OPTIONS 차단 대응을 위한 인증 인터셉터 임시 비활성화 및 트러블슈팅

[증상] 프론트엔드와 백엔드를 독립된 인프라로 분리하여 배포한 후, 로그인 및 특정 API 호출 시 브라우저 콘솔에 CORS 에러가 발생하며 모든 인증 기반 요청이 차단되는 결함이 확인되었습니다.

[원인 추적 및 임시 조치] 브라우저가 본 요청을 보내기 전 안전성을 검증하는 예비 요청(Pre-flight, OPTIONS 메서드)이 백엔드의 커스텀 인증 인터셉터(AuthInterceptor)에 걸려 차단되는 것이 원인이었습니다. 당시 배포 환경에서의 즉각적인 API 통신 성공 여부를 검증하기 위해, 백엔드 WebConfig에서 인증 인터셉터 등록 코드 자체를 주석 처리하여 임시로 비활성화하는 단기 우회책을 적용했습니다.

[추후 개선 포인트] 현재 상태는 전체 API의 인증 장벽을 임시로 걷어낸 조치이므로, 향후 보안 무결성을 확보하기 위해 인터셉터를 다시 활성화해야 합니다. 이를 위해 인터셉터 내부 진입점에서 HTTP OPTIONS 메서드만 선별적으로 통과시키는 예외 로직을 정교하게 구축하거나, Spring Security를 도입하여 필터 체인 전역 레벨에서 CORS와 Pre-flight 설정을 체계적으로 분리 및 통합 관리할 계획입니다.

CASE 03

특정 LLM 모델 부하 대처를 위한 하이브리드 Fallback 우회 시스템 구축

[문제] 메인으로 사용하는 Gemini 특정 모델의 서버가 순간적으로 바빠지거나 트래픽이 몰릴 때, 간헐적인 응답 지연 및 API 오류가 발생하여 서비스가 비정상적으로 중단되는 현상이 확인되었습니다.

[해결 및 Fallback 구현] 단일 AI 모델 의존으로 인한 리스크를 방지하기 위해 백엔드에 하이브리드 LLM 오케스트레이션 로직을 설계했습니다. Gemini API 호출 시 에러가 발생하면 즉각 OpenAI API로 요청을 자동 전환하는 우회(Fallback) 구조를 구축하여 서버 과부하 상황에서도 서비스가 끊김 없이 유지되도록 조치했습니다.

[결과] 모델 자체의 서버 불안정성으로 인한 단일 장애점을 제거했으며, 연동 테스트 과정에서 간헐적인 외부 서버 에러가 발생하더라도 사용자에게는 안정적인 분석 결과를 제공할 수 있게 되었습니다.

배운 점 및 성장

query_stats 단순 텍스트 출력을 넘어선 구조화된 데이터 설계의 가치

AI 기반 서비스를 구축할 때 줄글 형태의 비정형 답변은 일시적인 위로를 줄 수 있어도 유저의 근본적인 사고 구조를 변화시키는 통계적 자산이 되기 어렵다는 점을 배웠습니다. 비정형 데이터를 엄격한 JSON 스키마 규격으로 압축하여 가치 있는 구조적 데이터로 축적할 때 비로소 정교한 인지 왜곡 교정 시스템이 작동할 수 있음을 체득했습니다.

alt_route 인프라 제약 속에서의 주도적인 리스크 통제 및 비용 설계

트래픽 비용 리스크를 사전에 식별하여 시스템 프롬프트 키값을 다이어트하는 아이디어를 직접 실현하고, 분산 환경에서의 CORS/세션 불일치와 클라우드 API 장애 한계를 극복하기 위해 하이브리드 Fallback 아키텍처를 주도적으로 구축해 내며, 제약 조건 속에서 최선의 차선책을 도출하는 리스크 관리 역량을 길렀습니다.
