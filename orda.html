문제 해결과 기술적 고민

CASE 01

OSM 및 공공데이터 등산로 표준화 파이프라인의 좌표계 오류 및 병합 로직 결함


[증상] 시각화해서 확인했을 때 일부 등산로 선형이 한국 내륙을 벗어나 동해 바다 및 대만 인근에 찍히는 현상이 발생했습니다.

[원인 추적 및 해결] 처음에는 일부 데이터만 동해 바다에 나타나 병합 시 문제라고 생각했는데 문제가 있는 데이터의 출처가 모두 공공 데이터라는 것을 확인하고 좌표계 문제를 의심했습니다. 최종 원인은 원본 데이터가 GRS80 타원체 기반 TM 중부원점(20, 60) 투영 기준으로 설계되었음을 판명하고 코드 수정으로 바로 잡았습니다.

[예외 및 로직 고도화] 분석 중 발생한 음수 X축 값에 대해 False Easting 누락 오류로 오해하기 쉬웠으나, 실제 데이터 분포 조사 결과 백령도 지형의 예외적 특성임을 포착하여 수동 오보정을 방지했습니다. 또한 좌표계 정상화 직후 그동안 가려져 있던 'A단계 병합 로직 결함(공공데이터와 OSM 60% 중복 시 유효 속성을 이관하지 않고 OSM을 폐기하여 정보 손실 발생)'을 도출, 유효 속성 보존 가이드라인을 수립하고 통계 로그 보강을 완수했습니다.

CASE 02

서로 다른 산의 등산로 네트워크 결합 버그와 데이터 소스 커버리지의 한계

[증상] 등산로 네트워크 구축 단계에서 산과 산 사이의 경계가 무너지고 서로 다른 산의 등산로가 하나의 거대한 네트워크 그래프처럼 강제 연결되는 현상이 확인되었습니다.

[원인 추적 및 개선] build_network.py를 분석한 결과, NodeRegistry가 산 식별자(MNTN_CODE, mountain_name) 없이 (lon, lat) 좌표만으로 노드를 생성하고 있었습니다. 실제 데이터를 확인한 결과, 서로 다른 산의 동일 좌표에서 만나 하나의 노드로 처리되는 사례가 1,524개 확인되었습니다. 이를 해결하기 위해 노드 식별 기준을 (lon, lat, mountain_key)로 변경해 같은 좌표라도 산이 다르면 서로 다른 노드로 취급하도록 수정했습니다. 다만 노드 분리 이후 prune_dangling_edges()가 작은 고립 구간을 제거하는 과정에서 정상 공공 등산로까지 함께 삭제하는 문제가 발생했습니다. 이를 보완하기 위해 official/PUBLIC 데이터는 pruning 대상에서 제외하고, edge 병합 과정에서도 mountain_key가 유지되도록 수정했습니다.

[도메인 인사이트] OSM 데이터를 제거하고 공공데이터만 확인해 보니 계룡산 금수봉과 같은 유명 산의 등산로가 비어 있거나 충분히 제공되지 않는 경우가 확인되었습니다. 이를 통해 '다른 산이 연결된다'는 문제가 단순한 네트워크 버그가 아니라, 공공데이터만으로는 ORDA가 목표로 하는 산 커버리지를 제공하기 어렵다는 문제와도 연결되어 있음을 알게 되었습니다. 결국 실제 등산 코스가 더 풍부하게 포함된 GPX 기반 데이터로 방향을 전환했습니다. 다만 GPX 데이터 역시 공유 구간과 중복 선형이 많아 별도의 정제 작업이 필요하다는 점을 확인했고, 우선 실제 등산 테스트를 진행한 뒤 후속 정제를 진행하기로 결정했습니다.

CASE 03

리플레이 기능 구현을 위한 런타임 제어용 멀티 시간축 분리 설계


[문제] 단순 등산 기록 시간(actualElapsedSeconds)축만 활용해 리플레이를 재생할 경우, ORDA 서비스 특유의 시네마틱 연출(인트로 오버뷰, 시작점 포커스, 아웃트로 전체 경로 복귀)이 개입될 때 슬라이더의 타임라인 진행률, 카메라 뷰포트 상태, 마커 위치 연산이 통째로 꼬여버리는 구조적 한계가 존재했습니다.

[시간축 분리 구현] '실제 데이터 시간'과 사용자가 체감하는 '영상 제어 시퀀스 시간'의 도메인을 명확히 분리하여 두 개의 시간축을 분리해서 사용하는 구조를 도입했습니다.

  • 1) 시퀀스 시간축 (sequenceElapsedMs): 인트로(2.2초)/포커스(1.2초)/본 재생/아웃트로(1.8초)를 포괄하는 전체 런타임 축으로, 슬라이더 핸들러와 단계적 카메라 뷰 모드(intro-overview → focus-start → follow → outro-overview) 전환의 타임라인 기준으로 동작합니다.
  • 2) 실제 데이터 시간축 (currentReplaySeconds): GPS 트랙 로그 정보의 보간 연산, 진행 시간 틱(50ms 단위) 연산 및 정상 마커 노출 시점 연산(verifiedElapsedSec <= currentActualElapsedSeconds) 등 데이터 무결성 검증을 전담합니다.
CASE 04

대용량 DEM GeoTIFF 처리 과정에서 발생한 JVM OOM과 배포 환경 제약 대응


[문제] 실시간 GPS 위치를 초 단위로 처리해야 했기 때문에, 고도 보정을 위해 매 요청마다 Supabase 원격 스토리지에서 DEM 데이터를 읽어오는 방식은 네트워크 지연으로 인해 현실적으로 사용하기 어려웠습니다. 또한 대한민국 DEM GeoTIFF 파일은 약 300MB 크기로 Git의 단일 파일 업로드 제한(100MB)을 초과해 리포지토리에 직접 포함할 수 없는 상황이었습니다.

[원인 추적 및 대응] 이를 해결하기 위해 .dockerignore 설정을 조정해 GeoTIFF 파일만 Docker 빌드 과정에서 이미지 내부(/app/data/dem/korea_dem.tif)로 포함하도록 구성했습니다. 그러나 배포 후 DemService가 시작 시점에 GeoTIFF 전체를 메모리로 읽어오는 Eager Load 방식으로 동작하면서 기본 JVM 힙 메모리(Xmx400m)를 초과해 OutOfMemoryError가 발생했습니다.

[대응 및 개선 방향] Render 무료 플랜의 메모리 제약으로 배포가 어려워지자, 우선 현장 테스트 일정을 맞추기 위해 JVM 옵션을 -Xmx1200m, -Xms512m으로 조정하고 amd64 이미지를 재빌드해 GCP Cloud Run 환경으로 배포를 전환했습니다. 이 과정에서 단순히 메모리를 늘리는 것이 근본적인 해결책이 아니라는 점을 확인했고, 이후 등산로 데이터 정제가 마무리되면 Eager Load를 제거하고 타일 기반 분할과 Lazy Load 방식으로 리팩터링할 계획을 세웠습니다.

배운 점 및 성장

query_stats 수치적 지표를 넘어선 시각적 검증의 중요성

파이프라인 로그나 데이터베이스 카운트가 정상적인 '성공'을 가리키고 있더라도, 실제 공간 레이어에 올렸을 때 비로소 드러나는 위상적 왜곡과 좌표 이탈 현상을 목격했습니다. 지도 기반 서비스 엔지니어링은 텍스트나 숫자 지표에만 매몰되지 않고, 데이터 분포를 다차원적으로 '직접 눈으로 증명'해야만 완전한 무결성을 얻을 수 있음을 배웠습니다.

alt_route 기술적 제약 속에서의 현실적인 리스크 매니지먼트

데이터 소스의 한계(공공데이터 누락 및 GPX의 중복도)를 빠르게 포착하여 서비스 범위에 맞게 데이터를 재선택하고, 인프라 자원 부족(OOM)이라는 제약 속에서 현장 테스트 일정을 사수하기 위해 Docker 컨텍스트 활용, JVM 메모리 튜닝 및 GCP 마이그레이션과 같은 단계별 차선책을 리드해 내며 현실적이고 주도적인 리스크 관리 능력을 체득했습니다.

psychology AI를 활용한 단계별 개발 및 제어 전략

이번 프로젝트에서는 원하는 기능들을 구현하기 위해 조금 더 적극적으로 AI를 활용했습니다. 그러면서 아직 배우는 단계이다 보니, 한 번에 너무 많은 코드를 작성하고 나면 나중에 어디서 버그가 터졌는지 찾기도 어렵고 수정하기도 힘들다는 것을 배웠습니다. 그래서 구현하기 까다롭고 복잡한 기능일수록 작업을 아주 작게 쪼개고, 단계별로 꼼꼼하게 검증하는 저만의 안전한 개발 워크플로우를 구축하여 AI를 똑똑하게 사용하고 리스크를 통제했습니다.

STEP 01. 작은 단위로 계획 세우기
구현해야 할 복잡한 기능을 AI에게 먼저 설명하고, 이를 아주 작은 작업 단위로 나누어 계획을 세우게 했습니다. 코드를 무작정 짜기 전에 이 플랜이 제 의도와 맞는지 먼저 꼼꼼하게 검토했습니다.
STEP 02. Git Diff로 변경 내역 확인하기
하나의 작은 작업이 끝날 때마다 Git Diff를 열어 어떤 코드가 어떻게 변경되었는지 라인 단위로 살펴봤습니다. 자동 생성된 코드가 제가 생각한 방향과 일치하는지 제 수준에서 완벽히 이해하고 넘어갔습니다.
STEP 03. PR을 통한 팀원 교차 검증
이렇게 제 수준에서 1차 검증이 끝난 코드들을 모아 Pull Request를 올렸습니다. 혼자서는 놓칠 수 있는 제약 사항이나 예외 케이스들을 팀원들과 함께 다시 한 번 교차 검증하며 코드의 안정성을 높였습니다.
